## 引継ぎ指示(Grok Operator 向け)

あなたはこのワークスペースの Operator です。ルートの `AGENTS.md` と `.agents/skills/` に従い、
以下の 4 プランを**このハーネス自身の運用ルールで**実行してください。これはドッグフーディングです:
ハーネスの規律(packet → delegate → verify)を守れない場合、それ自体が発見としてレポート対象です。

各プランの実行プロトコル:

1. `context-loader` 相当の最小ロード(AGENTS.md / STATE.md / 対象プランのみ)で開始する。
2. プランを全文読んでから着手する。**Drift check を必ず最初に実行**する。
3. 設計判断が必要な箇所は Codex `read-only` に委譲する(Prompt Contract packet を
   `.agents/docs/packets/plan-NNN-design.prompt.txt` に書き、`scripts/delegate-codex.ps1` 経由)。
4. 実装は Grok(あなた)が行う。デフォルトルーティング通り。
5. 非自明な変更後は Codex `review` packet を必ず通す(AGENTS.md「Done means」参照)。
6. `verify-job` スキルのチェックリストを実行してから DONE を宣言する。
7. 下の Status 表と `.agents/STATE.md` を更新する。
8. プランごとに feature ブランチ + PR(直近の履歴に倣い `feat/...` / `docs/...` 形式、
   コミットメッセージは英語)。push / PR 作成前にユーザーへ確認する。

**STOP 共通条件**: 各プランの STOP conditions に加え、同一の失敗が 2 回続いたら
サーキットブレーカー(AGENTS.md F05/F17)を発動し、第 3 の変種を試さずユーザーへ報告する。

### PR レビューボット対応プロトコル(無限ループ防止付き)

PR 作成後に Codex bot 等のレビューが付いたら、次の**有界ループ**で対応する:

1. **検出**: reviews API と issue comments の**両経路**を確認する(bot の指摘は formal
   review ではなく comment で来ることがある)。login フィルタには `[bot]` を含める。
2. **トリアージ**(指摘ごとに 3 分類。全指摘に対し分類結果を PR コメントで返信する):
   - **fix**: 正当かつ当該プランの In scope 内 → 修正する
   - **defer**: 正当だが scope 外 → 修正せず、`plans/README.md` の Plan errata /
     follow-up に記録し、その旨返信する
   - **decline**: 一次情報(file:line)で裏取りして誤検出・好みの問題と判断 → 修正せず、
     裏取り結果を引用して返信する
3. **修正ラウンドの上限は 1 PR につき 2 回**。各ラウンドは「fix 分類の全修正 →
   ローカルテスト green → verify-job PASS → push」を 1 単位とする。
4. **打ち切り条件**(いずれかで即座にループを抜け、ユーザーへ報告して指示を待つ):
   - 2 ラウンド消化後も新規の fix 対象指摘が残っている
   - 同一指摘が修正後に再度指摘された(2 ストライク = F05/F17)
   - 指摘への対応が scope 外ファイルの変更を要求する
   - bot と自分の判断が矛盾し、一次情報でも決着しない
5. **禁止**: decline / defer への再返信の応酬(返信は指摘 1 件につき 1 回まで)。
   レビュー再依頼の連打。指摘対応を理由にした scope 拡大。
